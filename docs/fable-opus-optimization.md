# Opus 4.8 を Fable 5 に寄せる実運用ガイド（決定版）

Fable サブスクが使えなくなる前提で、`claude-opus-4-8` の出力品質と振る舞いを
`claude-fable-5` にできるだけ近づけるための実運用メモ。
最終照合: 2026-06-12（Anthropic 公式 docs に照合済み。出典は末尾）

## TL;DR

1. **設定**: `output_config={"effort": "xhigh"}` を明示（既定は high）、`thinking={"type": "adaptive"}` を明示（省略＝思考オフ）、`max_tokens` は 64K 開始。
2. **プロンプト**: 公式推奨スニペット（過剰計画抑制・過剰実装禁止・根拠付き進捗・越権禁止）＋ 仕様と意図を冒頭 1 ターンで全部渡す。
3. **構成**: フレッシュコンテキストの検証 sub-agent ＋ Batch API（50% 引）での best-of-N。同一予算なら **Fable 単発 1 回 ≒ Opus 4.8 Batch 約 4 案＋検証**。ここで逆転を狙う。

## 0. 前提と仕様（2026-06 時点）

素の能力差は実在する（長期自律、複雑タスクの一発正答率、サブエージェント運用、バグ検出の再現率で Fable 優位）。ただし:

- **挙動の差**（サブエージェントを起動しない、ツールを使い渋る、対話の小出しで効率が落ちる等）は、公式が手順を示している範囲でプロンプトと構成で詰められる。
- **能力の差**は、半額を生かした「N 回生成 → 別コンテキスト検証者で選抜」で部分的に買い戻せる。
- 土俵の置き方: 公式推奨は **Fable = high 開始、Opus 4.8 = コーディング/エージェントで xhigh 開始**。比較すべきは「Opus 4.8 @xhigh vs Fable @high」。
- Fable 自身も cyber / bio / 思考抽出系の分類器が発火すると Opus 4.8 へフォールバックする設計。Opus 4.8 は「劣化版」ではなく公式の受け皿でもある。

| 項目 | Opus 4.8 | Fable 5 |
|---|---|---|
| Model ID | `claude-opus-4-8` | `claude-fable-5` |
| 価格 (in / out per MTok) | $5 / $25 | $10 / $50 |
| Batch API（50% 引） | $2.50 / $12.50 | $5 / $25 |
| キャッシュ (read / 5分write) | $0.50 / $6.25 | $1 / $12.50 |
| context | 1M（標準価格・βヘッダ不要）※1 | 1M（標準価格） |
| max output | 128K | 128K |
| thinking | adaptive を**明示**（省略＝思考オフ） | 常時オン（設定不要、`disabled` は 400） |
| effort | low / medium / high（既定）/ xhigh / max | 同左（推奨開始は high） |
| 廃止済み（送ると 400） | temperature・top_p・top_k（非デフォルト値）、`thinking.budget_tokens`、assistant prefill | 同左 |

- ※1 Microsoft Foundry のみ 200K。
- ※2 Opus 4.7 以降は新トークナイザ。同一テキストが旧世代比で最大 ~35% 多くトークン化され得る（コスト見積もり時に注意）。
- ※3 Fable 5 は 30 日データ保持が必須で ZDR 不可。ZDR 要件があるならそもそも Opus 4.8 一択。

## 1. 設定レベル（必須・即効）

| レバー | 設定 | 理由 |
|---|---|---|
| effort | `output_config={"effort": "xhigh"}` | 既定は high。コーディング/エージェントは xhigh 明示が公式推奨。**max は xhigh で伸び代を eval 確認できた難所のみ**（逓減・overthinking のリスク）。effort は思考だけでなくツール呼び出し回数・説明量まで含む全トークンに効く |
| thinking | `thinking={"type": "adaptive"}` を明示 | 省略すると思考オフ（Fable は常時オン）。推論サマリが要るなら `"display": "summarized"`（既定は `omitted`） |
| max_tokens | xhigh / max では 64K 開始で調整 | 思考＋ツール＋本文の合計上限。128K 級の長出力は `.stream()` ＋ `get_final_message()` 必須 |
| prompt caching | 安定プレフィックス（system / tools）に `cache_control` | read は 0.1x（$0.50/MTok）で **5分キャッシュなら 1 ヒット目から黒字**（write 1.25x）。最小キャッシュ長は Opus 4.8 で 4,096 トークン（Fable 5 / Sonnet 4.6 は 2,048） |
| 指示の途中更新 | mid-conversation system message（4.8 新機能） | 履歴を作り直さず `role:"system"` をユーザーターン直後に挿入 → 過去ターンのキャッシュを保ったまま方針変更できる |

```python
client.messages.create(
    model="claude-opus-4-8",
    max_tokens=64000,                      # xhigh/max では 64K 開始、長出力はストリーミング
    thinking={"type": "adaptive", "display": "summarized"},
    output_config={"effort": "xhigh"},     # max は eval で headroom を確認できた場合のみ
    system=[{"type": "text", "text": SYSTEM,
             "cache_control": {"type": "ephemeral"}}],
    messages=[...],
)
```

⚠️ temperature / top_p / top_k（非デフォルト値）、`thinking.budget_tokens`、assistant prefill はいずれも 400。SDK の型定義にはフィールドが残っているため**型チェックは通るがサーバ側で拒否される**点に注意。深さの制御は effort のみ。

## 2. プロンプト様式（無料で効く・公式準拠）

公式「Prompting Claude Fable 5」のスニペットは Fable の挙動矯正用だが、同系統の挙動は Opus 4.8 にもあり、ほぼそのまま流用が効く。ここが一番コスパが高い。

**冒頭 1 ターンに全仕様＋意図を渡す。** Opus 4.8 は対話型だとユーザーターンのたびに再思考が増えてトークン効率（時に性能も）が落ちる公式特性があるため、小出しは二重に損。「誰のため・それで何が可能になるか」という意図を添えると性能が上がる（公式記載）。

コピペ用スニペット（system / カスタム指示向け。必要なものだけ選ぶ）:

```
過剰計画の抑制:
行動できる情報が揃ったら行動する。会話で確定済みの事実を再導出せず、決定済みの
判断を蒸し返さず、選ばない選択肢をユーザー向け文面で羅列しない。判断するときは
網羅列挙ではなく推奨を1つ出す（思考ブロック内はこの限りでない）。
```


```
過剰実装の禁止:
タスクが要求する以上の機能・リファクタ・抽象を足さない。バグ修正に周辺の掃除は
不要。起こり得ないシナリオのエラーハンドリングやフォールバックを書かず、検証は
境界（ユーザー入力・外部API）のみ。将来要件を先回りせず、最小で確実に動くものを出す。
```


```
進捗主張の根拠付け:
進捗報告の各主張を、このセッションのツール結果に照合してから述べる。未検証なら
未検証と明示する。テストが落ちたら出力を添えてそう言い、完了・検証済みのことは
ためらわず言い切る。
```


```
越権の禁止:
ユーザーが問題の説明・質問・思考の整理をしているだけのときは、所見を返して止まる。
修正は依頼されてから。システム状態を変えるコマンド（削除・設定変更・再起動）の前に、
根拠がその操作を本当に支持しているか確認する。
```


```
停止基準:
ユーザー確認で止まるのは、破壊的・不可逆な操作、実質的なスコープ変更、ユーザー
しか持たない情報が必要なときだけ。該当したら質問してターンを終える。
```


```
結論先行:
最初の一文が「何が起きたか／何が分かったか」に答えること。短くするのは圧縮
（断片・矢印連鎖・略語）ではなく、読者の次の行動を変えない詳細を落とす取捨選択で行う。
```

**Opus 4.8 固有の差分を埋める 2 点:**

- **文字通りに解釈する**: 指示を黙って一般化しない。広く適用させたい指示はスコープを明示する（例:「最初のセクションだけでなく全セクションに適用」）。
- **ツールを使い渋る**: 推論で済ませる傾向がある。第一レバーは effort を上げること。足りなければ「いつ・なぜそのツールを使うか」を tool description と system に明記する。

**指示はまず減らす。** 旧モデル向けの矯正指示（「3 ツールごとに進捗要約せよ」等）は 4.8 では不要〜逆効果になりうるので、外して素の挙動を確認してから足す。「CRITICAL: YOU MUST …」調も同様。

## 3. エージェント構成（モデル差をシステム差で埋める）

Fable の優位（並列 sub-agent の積極運用・長期自律・メモリ活用）は、4.8 では「明文化」で引き出す。

- **サブエージェント起動を明文化。** 4.8 は既定で起動が控えめだが、プロンプトで制御可能（公式）。例:

```
  単独で完了できる作業に subagent を立てない。複数アイテムへのファンアウトや
  複数ファイルの読込は、同一ターンで複数の subagent を起動する。独立サブタスクは
  委譲し、自分は手を止めずに進める。逸脱やコンテキスト不足があれば介入する。
```

- **非同期＋長命 subagent。** 各 subagent の完了を待たずに本ループを進める。コンテキストを保持する長命 subagent は cache read でコストが下がり、最遅タスクに律速されない。低 effort で回す subagent には「このタスクは多段推論を要する。回答前に熟考すること」を 1 行足す（公式）。
- **フレッシュコンテキストの検証者。** 自己批評より、経緯を知らない別コンテキストの検証 sub-agent が効く（公式）。長期タスクには「[X] 間隔で、仕様に照らして subagent により成果を検証する方法を確立せよ」を仕込む。検証者には**仕様と成果物だけ**を渡し、生成側の思考や経緯は渡さない。
- **メモリ（.md）。** 1 ファイル 1 教訓＋冒頭 1 行サマリ。リポジトリや履歴に既にあることは書かない、重複は新規作成でなく更新、誤りと分かったら削除（公式推奨形式）。4.8 は自発的に手を伸ばしにくいので、使用条件を tool description と system に明記する。
- 超長セッションは docs の Compaction / Context editing を参照。Claude Code の ultracode 相当（xhigh＋マルチエージェント常設許可）を API で組むなら公式「Build an orchestration mode」が雛形。

## 4. コスト再投資（同一予算で Fable 単発超え）

価格差の算数: Fable 通常 $10/$50 に対し、Opus 4.8 通常は 1/2、**Batch API なら 1/4（$2.50/$12.50、入出力とも 50% 引）**。Fable 単発 1 回の予算で、非対話タスクなら Opus 4.8 を約 4 案生成してもまだ検証分が残る。

- **重要・非対話タスク**: Batch で N=2〜4 生成（共通プレフィックスはキャッシュ併用可）→ フレッシュ検証者（medium〜high で足りることが多い）が仕様照合で選抜・合成 → 必要なら xhigh で仕上げ。
- **対話・レイテンシ重視**: xhigh 単発＋検証パス 1 回。
- キャッシュで高 effort 運用の固定費（長い system / tools）を 0.1x に圧縮。
- 注: **Fast mode は使わない**。速度のための課金（4.8 で $10/$50）で品質は変わらず、品質パリティという本ガイドの目的に寄与しない。

## 5. 検証（思い込みを排す）

- 自前の小さな eval set を用意し、medium / high / xhigh（必要なら max）×（素プロンプト / 本ガイド適用）でスイープして実測する。effort の寄与は歴代 Opus で最大級（公式）なので、ここをサボらない。
- レビュー系タスクで recall が落ちたら、能力低下ではなく「自己フィルタ」（指示を忠実に守って軽微な指摘を黙る挙動）を疑い、下記 C の網羅ファースト指示に切り替える。
- 効くものだけ残す。まず指示を減らして素で試すのが定石。

## 汎用チャット（claude.ai など）での使い方（コピペ用）

effort / max_tokens は API 限定。チャットでは ①使える最上位モデルを選ぶ ②思考（拡張思考）を有効化できる場合はオン ③以下を貼る、で挙動を寄せる。

### A. ベース指示（カスタム指示 / Projects / 会話の最初に貼る）

```
できるだけ深く、段階的に検証しながら考えてください。以下の方針で動いてください。

- 行動できる情報が揃ったら行動する。確定済みの事実を再導出せず、決定済みの判断を蒸し返さず、選ばない選択肢を羅列しない。判断するときは網羅列挙ではなく推奨を1つ出す。
- 要求された以上の機能・リファクタ・抽象を足さない。バグ修正に周辺の掃除は不要。起こり得ないケースの防御コードは書かず、検証は境界（ユーザー入力・外部API）のみ。最小で確実に動くものを出す。
- 進捗や結果の主張は、実際に確認できた事実だけを述べる。未検証なら「未検証」と明示する。テストが落ちたら出力を添えてそう言う。
- 私が問題を説明している・質問している・考えを整理しているだけのときは、所見を返して止まる。修正や変更は依頼されてから。確認のために止まるのは、破壊的・不可逆な操作、スコープ変更、私しか知らない情報が必要なときだけ。
- 結論から書く。最初の一文で「何が起きたか／何が分かったか」に答える。短くするのは圧縮ではなく取捨選択で。強い前置き（「重要」「必ず」等）は不要。
```

### B. タスク開始テンプレ（最初のメッセージで仕様を全部渡す）

```
【目的】何を達成したいか
【背景】誰のためで、それによって何が可能になるか
【成果物】出力の形式・粒度
【制約】守るべき条件・使う技術・やってはいけないこと
【完了の定義】どうなれば「できた」か（可能なら検証方法も）
【入力】関連ファイル・既存コード・参考資料（あるものは全部このターンで）

上記を踏まえて、まず方針を1つ提案し、問題なければそのまま実装まで進めてください。
不明点は着手前に1つのメッセージにまとめて質問してください。
```

### C. 検証・別案出し（重要な回答のフォローアップに貼る）

```
いまの回答を、別人のレビュアーとして批判的に検証してください。
仕様・制約に照らして、誤り・抜け・リスクを「すべて」箇条書きで挙げてください。
この段階では重要度や確信度で間引かないこと（見逃しより過剰検出がよい）。
各指摘に確信度と推定重要度を添え、最後に重要度順に並べ替えたうえで、
必要なら修正版を出してください。
```


```
別の有力なアプローチを2つ、それぞれの長所・短所付きで出してください。
そのうえで、この要件に最も適したものを1つ推奨してください。
```

### D. 新規チャット検証（フレッシュコンテキストの再現）

API の「別コンテキスト検証者」をチャットで再現する手順。**新しいチャット**を開き、(1) 仕様・制約 (2) 成果物 だけを貼る（経緯・説明・案の出自は貼らない）。そのうえで:

```
あなたはこの成果物の作成に関与していないレビュアーです。
添付の仕様・制約に照らして、誤り・抜け・リスクをすべて挙げてください。
重要度や確信度で間引かず、各指摘に確信度と推定重要度を添えてください。
最後に「採用可 / 修正後採用 / 作り直し」の判定を理由付きで出してください。
```

best-of-N の比較も同じ手順で、案 A/B/C をラベルだけ付けて貼り、
「仕様に最も適合する案を1つ選び、他案から取り込むべき点があれば挙げよ」と依頼する。
