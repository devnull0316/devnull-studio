# 意思決定・調査ログ

プロダクトの方向を決めた根拠の記録。受験で「なぜこの形にしたか」を語るための素材でもある。

---

## D1. 「iOS標準の予測変換そのもの」を切り替える案 → 不採用
**事実**：iOSアプリ／キーボード拡張は **Apple純正キーボードの学習データ（予測変換・オートコレクト・変換履歴）に一切アクセスできない**（読み書き不可・サンドボックス隔離）。ユーザー辞書（テキスト置換）を `UILexicon` で読めるだけ。
**結論**：「標準の予測変換を名前付きで持ち運ぶ」は技術的に不可能。**自前のカスタムキーボードを作り、その中の学習データを切り替える**形に変更。
出典: Limitations of custom iOS keyboards (inFullMobile/Medium), Apple Developer Docs。

## D2. 自前キーボードの容量・メモリ問題 → 解決可能（先行事例あり）
**事実**：キーボード拡張は **約30〜60MB でクラッシュ**（端末依存）。日本語かな漢字変換辞書は数十万語で巨大。
**先行事例 azooKey**（OSS, Swift, Apache-2.0）：**LOUDS（簡潔データ構造）＋頭文字ごとの分割遅延ロード**で、30万語をランタイム **約20MB台**に収めて実装済み。
**結論**：本格変換を載せる段階では LOUDS＋分割ロードを採用。ただし**製品の新規性は変換品質ではなく「学習データの切替・持ち運び」**なので、まずそこを作る。
出典: azooKey をOSSにした (Zenn), Swiftで日本語入力 (Qiita), google/mozc README, Apple Developer Forums。

## D3. 辞書は2層に分かれる → 持ち運ぶのは「学習データ層」だけ
**洞察**：①システム辞書（大・固定・共通でよい）と ②学習データ＋ユーザー辞書（小・可変）。
**結論**：プロファイルとして着脱・同期するのは小さい②だけ。重い①は据え置き。**容量問題を回避しつつコンセプトが成立**。→ エンジンの `LearningStore` がこの②に相当。

## D4. azooKey：フォーク vs 自作 → 当面エンジンは自作、UIは参照
**トレードオフ**：フォーク＝最速で動く／自作＝完全オリジナル・IPと受験の物語が強い・OSS化を強制されない。
**現状の判断**：**エンジンは自作**（`PersonaEngine`）。azooKeyは LOUDS辞書やキーボードUIの**参照実装**として読む。最終確定は M2 で再評価。

## D5. Windows で作れるか → エンジンは可、iOSアプリは不可
**事実**：Swift言語は Windows公式ツールチェーンでビルド可。だが **UIKit / iOS SDK / コード署名は macOS 専用**。Flutter等でも最終ビルドは裏で Xcode を呼ぶ＝macOSは消せない。
**結論**：**エンジン層（Foundationのみ）を Windows で開発・テスト**し、**iOSシェルだけクラウドMac**で組む2層構成（→ `ARCHITECTURE.md`）。本リポジトリの engine は実際に Windows/Linux で全テスト通過。
出典: Swift.org (Swift on Windows), Apple/Codemagic docs, Swift Forums。

## D6. テストの切り分け → Windowsはエンジン検証、体験テストは実機
**注意点**：エンジンを Windows で「友達に使わせる」には使い捨てのWindows UIが要り、しかも得られるのは変換精度の話だけ。**製品の核（人格レイヤー着脱の体験・心理）は実機でしか取れない**。
**結論**：エンジンの正しさ＝**自動テスト**で担保。友達による体験テストは **TestFlight（実機）**で。

## D7. Windows上で「本物のiOSキーボード」を再現 → 不可
**事実**：iOS Simulator は macOS専用。Windows用にiOSアプリ拡張を動かせる正規エミュレータは無い（Appetize等のクラウドSimulatorもキーボード拡張は深い統合ゆえ不向き）。
**結論**：本物の確認はクラウドMacのSimulator か TestFlight実機。Windowsに再現UIを作り込むのは労力対効果が低く、やらない。

## D8. 配布と匿名性 → 受験はTestFlightクローズド、公開は法人化後
**事実**：個人アカウントは本名が販売者名に公開・変更不可。AltStore PALも匿名でない。→ 「公開×匿名×非組織」は両立不可。
**結論**：受験フェーズは **TestFlightクローズド**（公開リスティングを作らない＝本名バレ回避、$99は必要）。公開は **2027-04 の法人登録（D-U-N-S＋屋号）後**に屋号名義で。（→ `DISTRIBUTION.md`）

## D9. ゴール設定 → 市場制覇でなく「合格」に最適化
**論点**：「実装が簡単で市場が空いている」＝参入障壁が低い＝秘密保持はmoatにならない。本当のmoatは着想・実行・継続・知名度。
**結論**：受験という締め切りに全振り。OSS化の是非は**受かるまで決めない**（先送り可能なら先送り）。AIで誰でも作れる時代、希少なのはコードでなく**着想＋実ユーザー＋考察**。

## D10. 指標 → スター数でなく「実ユーザー＋洞察」
**論点**：スターは取得が遅く・難しく・盛れる（心理学科の評価でむしろ逆効果）。
**結論**：追う指標は **少数でも本物のユーザーと、そこから得た心理学的洞察**（例：人は痕跡を「消す」より「隠して戻せる」方を好むか）。質的データを受験にぶつける。

---

## 心理学的フレーミング（受験の軸）
技術のmoatより強い差別化。人は文脈ごとに自己を切り替える（見せる顔／隠す顔）、AIに本音を吐きその痕跡を消したくなる心理、「人格レイヤーの着脱」＝自己の多面性・防衛機制・プライバシー。
語り方：「キーボードを作った」ではなく **「人が自己を隠す/切り替える行動に着目し、技術で扱えるようにした」**。臨床心理学のテーマと直結。
